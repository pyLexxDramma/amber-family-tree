# Что нужно от бэкенда

Фронт готов к подключению реального API: все экраны ходят через контракт `AngeloApi` (см. [api-contract.md](api-contract.md)). Чтобы подменить mock на бэк, нужны от вас:

## 1. Базовый URL API

- Один базовый адрес для всех запросов (например `https://api.angelo-test.ru/v1` или прокси с фронта).
- На фронте используется переменная окружения `VITE_API_BASE` (без слэша в конце).

## 2. Авторизация

- Как передавать токен/сессию: заголовок (например `Authorization: Bearer <token>`), cookie или иной способ.
- Как фронт получает токен после успешного логина/регистрации (тело ответа, заголовок).
- Поведение при 401: один формат ошибки, чтобы фронт мог перенаправить на логин или обновить токен.

## 3. Формат ответов

- JSON с полями, совместимыми с типами из контракта (см. `src/types/index.ts`). Если названия полей на бэке отличаются (snake_case и т.п.) — прислать маппинг или описание.
- Единый формат ошибок (например `{ code, message }`) и HTTP-коды для типичных случаев (400, 401, 404, 422, 500).

## 4. Эндпоинты под контракт

Во фронте уже подготовлен HTTP-клиент (`src/integrations/realApi.ts`), который ожидает такие пути (относительно `VITE_API_BASE`):

| Метод контракта | HTTP | Путь | Тело/параметры |
|-----------------|------|------|----------------|
| feed.list | GET | /feed | query: limit, offset, authorId, topicTag |
| feed.getById | GET | /feed/:id | — |
| family.listMembers | GET | /family/members | — |
| family.getMember | GET | /family/members/:id | — |
| auth.login | POST | /auth/login | { identifier } |
| auth.register | POST | /auth/register | { identifier } |
| auth.me | GET | /auth/me | — |
| profile.getMyProfile | GET | /profile/me | — |
| profile.updateMyProfile | PATCH | /profile/me | Partial<FamilyMember> |
| profile.listMyMedia | GET | /profile/media | — |

Если бэк будет использовать другие пути или имена полей — нужно прислать маппинг, фронт поправит в `realApi.ts`. Токен авторизации фронт передаёт в заголовке `Authorization: Bearer <token>` (ключ в localStorage — `angelo_token`, сохранять после логина/регистрации можно по договорённости).

## 5. Регистрация и логин

- Текущий поток на фронте: пользователь вводит телефон/email → «Получить код» → переход на экран ввода кода. Нужна спецификация: какой эндпоинт запрашивает код, какой отправляет код и возвращает токен/сессию, коды ошибок (не найден, неверный код и т.д.).

## 6. Медиа (загрузка файлов)

- Как загружать файлы: отдельный endpoint, multipart/form-data, максимальный размер.
- Как в ответе возвращаются URL загруженных файлов и привязка к публикации или профилю (чтобы фронт мог вызывать `listMyMedia` и отображать медиа).

---

После получения спецификации и базового URL фронт подключает реальный API в одном месте (`src/integrations/realApi.ts`), без правок по экранам.
