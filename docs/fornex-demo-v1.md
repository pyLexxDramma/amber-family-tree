# Fornex v1: Демо vs Реальная работа (Angelo)

Дата: 2026-03-12  
Проект: Angelo — Amber Family Tree  

## 1) Где “живёт” приложение на Fornex

- **Reg.ru**: только DNS (A‑записи указывают на VPS).
- **Fornex VPS**: фронт, бэкенд, БД, MinIO (S3).
- **Nginx на VPS**:
  - раздаёт фронт (статические файлы `dist/`)
  - проксирует **`/api/*` → backend (8000)**
  - проксирует **`/angelo-media/*` → MinIO (9000)** (для presigned upload через тот же домен)

## 2) Режимы работы в проде (на одном и том же домене)

Есть 2 “пользовательских” режима, переключаются без пересборки (runtime), через `localStorage`.

### 2.1. Демо‑режим (для ознакомления)

**Как включить**
- На главной (`/`) нажать **маленькую кнопку “Демо”**.
- Или в приложении: `Settings` → **“Включить демо”**.

**Что видит заказчик**
- Все основные экраны выглядят “живыми” и заполненными:
  - лента, публикации, семья, дерево, медиа — **данные из моков**
  - изображения/аватары берутся из ассетов проекта (локально, без внешних сервисов)
- В дереве, если у участника нет `avatar`, показываются демо‑аватары (fallback).

**Что при этом НЕ является “реальной работой”**
- Нет сохранения/доступа к пользовательским данным на сервере.
- Любые действия в демо работают как прототип (моки), кроме отдельных мест, если их специально “выводили” в real.

### 2.2. Реальный режим (не моки)

**Как включить**
- На главной (`/`) нажать **“Войти”** или **“Создать аккаунт”** — демо выключается.
- Или в приложении: `Settings` → **“Выйти из демо”**.

**Что происходит**
- Авторизация идёт через реальный API:
  - `POST /api/auth/send-code`
  - `POST /api/auth/verify` → фронт сохраняет `access_token` в `localStorage` (`token`)
  - дальнейшие запросы идут с `Authorization: Bearer <token>`
- Медиа‑сценарий работает с реальным бэкендом:
  - `POST /api/media/presign` → `PUT` файла в MinIO → `POST /api/feed` с `media_keys`
  - “Мои медиа”: `GET /api/profile/me/media`

## 3) Приоритет переключателей (что сильнее)

1) **Runtime demo‑mode** (`localStorage['angelo-demo-mode']=1`) — самый высокий приоритет: включает полный `mockApi`.
2) `VITE_USE_MOCK_API=true` — полностью `mockApi` (для локальной разработки/скринкастов).
3) `VITE_DEMO_MOCK_DATA=true` — гибрид (реальный auth, но моки части данных).
4) По умолчанию — полностью `realApi`.

На Fornex целевой сценарий для пилота:
- **показываем демо** кнопкой на Welcome/через Settings;
- **реальная работа** при входе/регистрации.

## 4) Что уже реализовано под Fornex (по факту)

- **Единый домен + `/api`**: фронт ходит на `/api/*`, Nginx проксирует в backend.
- **Presigned upload без CORS**:
  - публичный endpoint для браузера = `S3_PUBLIC_ENDPOINT_URL=https://<domain>`
  - upload URL вида `https://<domain>/angelo-media/<bucket>/<key>?X-Amz-...`
- **Демо‑переключение в UI**:
  - Welcome: маленькая “Демо”, большие “Войти/Создать аккаунт”
  - Settings: “Включить демо” / “Выйти из демо”
- **Дерево “от лица пользователя”**:
  - сверху видно “Вы вошли как …”
  - дерево строится от текущего пользователя
  - переход на Таймлайн/Медиа по клику на вкладку сразу

## 5) Вопросы от 10.03.2026 — ответы и статус реализации

Ниже — фиксация: **вопрос → ответ (как вы видите) → статус в текущем прототипе/коде**.

### 5.1. Регистрация и онбординг

- **Онбординг после SMS/email**
  - **Ответ**: минимум обязательных данных → проще всего первое действие: создать пост/загрузить фото → затем пригласить участника.
  - **Статус**:
    - UI: есть `Onboarding`, есть “Создать” и загрузка медиа.
    - Бэк: auth flow минимальный (для пилота можно усложнять).
    - **Открыто**: чёткий список “минимум обязательных полей”, что именно сохраняем на онбординге.

- **Создаёт ли пользователь “семью” или входит по инвайту**
  - **Ответ**: “общей семьи как сущности” нет; у каждого пользователя свой набор контактов и своё дерево; дерево может быть уникальным.
  - **Статус**:
    - В текущем бэке **есть** сущность `Family` и привязка `family_id` (исторически из бэкенд‑кода).
    - **Конфликт**: текущая реализация бэка против заявленного подхода “без семьи как общей сущности”.
    - **Нужно решение**: либо оставляем `Family` как “технический контейнер аккаунта/подписки”, либо убираем/рефакторим модель под “персональные деревья”.

- **Несколько семей у одного человека в пилоте**
  - **Ответ**: не предполагается как общая сущность.
  - **Статус**: требует решения на уровне модели данных (см. пункт выше).

### 5.2. Администрирование

- **Роли в MVP (админ/член/суперадмин/гость)**
  - **Ответ**: владелец контента vs читатель; “админ” = владелец подписки, который раздаёт платные места.
  - **Статус**:
    - В данных уже есть `role`, но логика прав/подписок пока прототипная.
    - **Открыто**: матрица прав (кто что может редактировать/видеть) и роли “гость/суперадмин” (нужны ли реально в MVP).

- **Кто может приглашать**
  - **Открытый вопрос**: только владелец подписки или все участники (с подтверждением)?
  - **Статус**: UI инвайта есть, нужна политика/правила.

### 5.3. Контент / медиа

- **Поддерживаемые форматы**
  - **Ответ**: популярные фото/видео/аудио (iPhone/Android/Win/Mac) + PDF.
  - **Статус**:
    - Технически: presign/PUT работает для любых `Content-Type`.
    - UI: выбор файлов и загрузка работают (пакетно).

- **Ограничения на размер**
  - **Ответ**: есть ограничения в BRD для MVP; в масштабе хочется снять (4K 60fps и т.п.).
  - **Статус**:
    - Фронт: лимит сделан настраиваемым (`VITE_MAX_UPLOAD_MB`, дефолт 1024MB).
    - Nginx: нужно выставлять `client_max_body_size` под целевые размеры.

- **Версии изображений (original/preview/thumb)**
  - **Ответ**: original хранить и скачивать, thumb для плитки, preview для ленты/публикации.
  - **Статус**:
    - Сейчас хранится 1 объект (без генерации превью/thumbnail).
    - **Открыто**: сервис генерации (в бэке), хранение ссылок и политика кеширования.

- **Пакетная загрузка**
  - **Ответ**: желательна.
  - **Статус**: в UI реализована мультизагрузка (несколько файлов).
  - **Открыто**: лимит на количество файлов в пакете/параллелизм.

- **Лимиты: только на общий аккаунт (5 чел → 100 ГБ)**
  - **Статус**: пока не реализовано (ни квота, ни отображение прогресса).

- **F‑1A “любое количество лайков от одного пользователя”**
  - **Ответ**: допускается больше 1 лайка.
  - **Статус**:
    - Сейчас бэк делает “1 лайк = 1 запись” (классический вариант).
    - **Нужно решение**: если реально нужно “много лайков”, меняем модель на счётчик/события.

### 5.4. Семейное дерево

- **Референсы и вид дерева**

- **Редактирование связей (F‑23)**

  - **Статус**: пока связи представлены базовыми типами; расширение — после фикса требований.
