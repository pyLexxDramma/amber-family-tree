# Спайк: мессенджер — свой vs встраиваемый (ANGELO)

Цель: дать заказчику основу для решения по MVP из видения (семейные чаты, медиа, реакции, звонки; возможен сторонний клиент для проверки гипотезы).

## Текущее состояние в репозитории

- REST API: `GET/POST /api/messages/with/{member_id}` — только **текст 1:1** в рамках одной семьи, без вложений, без комнат, без доставки в реальном времени.
- Фронт: диалог с членом семьи, список чатов как **заход из списка семьи**, не отдельный продуктовый «Мессенджер».

## Вариант A: развивать свой мессенджер

| Плюсы | Минусы |
|-------|--------|
| Полный контроль данных, приватность, модель «только родные» | Долго: WebSocket/long poll, медиа в чате, пуши, модерация, E2E если понадобится |
| Единый аккаунт и RBAC с деревом/событиями | Звонки WebRTC — отдельная подсистема (TURN/STUN, сигналинг) |
| Нет зависимости от блокировок внешних сервисов | Нагрузка на команду и инфраструктуру |

**Ориентир по этапам:** текстовые групповые чаты → вложения (через уже существующий S3 presign) → реакции/ответы → **позже** звонки.

## Вариант B: встраиваемый мессенджер / готовый SDK

| Плюсы | Минусы |
|-------|--------|
| Быстрее выйти на «живое общение» для проверки удержания | Данные и юридическая рамка у третьей стороны |
| Звонки и доставка часто «из коробки» | Сложнее сквозные сценарии (событие из чата, права «двух семей») |
| Меньше кода у вас | Риск несоответствия позиционированию «приватное семейное пространство», если бренд чужой |

**Риск из ТЗ:** тыловая формулировка «тестовые пользователи не должны знать» о встраивании — это **продуктово-юридический** вопрос (согласие, политика, этика), не только технический.

## Вариант C: гибрид

Короткий этап на внешнем SDK для валидации гипотезы удержания, параллельно проектирование своего протокола и миграция контактов/истории при готовности B2.

## Рекомендация для заказчика

1. **Зафиксировать MVP-мессенджер письменно:** только семья из древа или шире; группы обязательны или нет с первого релиза; звонки в MVP или нет.
2. Если срок критичен и допустима зависимость — **рассмотреть B** с явным раскрытием в оферте/онбординге (скрывать от пользователей нельзя без правовой оценки).
3. Если приоритет — **контроль и единая модель доступа с будущими «Событиями»** — **A** с поэтапной дорожной картой; первый шаг после текущего API — **групповые чаты + медиа** без звонков.

Технический долг текущего API: потребуется модель **комнат** (личные/групповые), участники, sequence сообщений, опционально вложения со ссылками на объекты хранилища.
