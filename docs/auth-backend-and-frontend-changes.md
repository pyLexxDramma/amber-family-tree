# Авторизация: бэкенд и доработки фронта

Документ описывает **систему авторизации на бэкенде** и **изменения на фронте**, необходимые для корректной работы входа при `VITE_USE_MOCK_API=false` (реальный API).

---

## 1. Система авторизации на бэкенде

### 1.1. Схема входа

Используется **двухшаговый OTP-подобный поток** (реальная отправка SMS/email не реализована):

| Шаг | Метод и путь | Назначение |
|-----|--------------|------------|
| 1 | **POST** `/api/auth/send-code` | Запрос кода. Тело: `{ "identifier": "email или телефон" }`. Ответ: `{ "sent": true }`. |
| 2 | **POST** `/api/auth/verify` | Подтверждение кода. Тело: `{ "identifier": "...", "code": "код" }`. Создаёт или находит пользователя, возвращает JWT и данные пользователя. |

Дополнительно:

- **GET** `/api/auth/me` — возвращает текущего пользователя. Требуется заголовок **`Authorization: Bearer <token>`**.

Эндпоинтов **POST** `/api/auth/login` и **POST** `/api/auth/register` на бэкенде **нет**.

### 1.2. Токены (JWT)

- Библиотека: `python-jose`.
- В payload: `sub` — ID пользователя (UUID), `exp` — время истечения.
- Алгоритм и параметры задаются в конфиге:
  - `JWT_SECRET` — секрет для подписи (в `.env.example`: `change-this-secret-in-prod`).
  - `JWT_ALGORITHM` — по умолчанию `HS256`.
  - `JWT_ACCESS_TOKEN_EXPIRE_MINUTES` — срок жизни токена (в примере: 10080 минут, 7 дней).

### 1.3. Проверка запросов

- Защищённые эндпоинты зависят от `get_current_user` (`app/core/security.py`).
- Заголовок **`Authorization: Bearer <token>`** обязателен; при отсутствии или невалидном токене возвращается **401 Unauthorized**.
- Защищены: `/api/auth/me`, все роуты в `feed`, `family`, `profile`, `media`.

### 1.4. Ответ verify

Эндпоинт **POST** `/api/auth/verify` возвращает JSON:

```json
{
  "access_token": "<JWT-строка>",
  "token_type": "bearer",
  "user": {
    "id": "...",
    "member": { ... },
    "subscription": { "planId": "free", "usedPlaces": 0, "expiresAt": "2026-12-31" }
  }
}
```

Поле `user` приведено к camelCase для фронта.

## 2. Текущее поведение фронта (реальный API)

При `VITE_USE_MOCK_API=false`:

1. **Экран входа / регистрации:** по кнопке «Получить код» вызывается `api.auth.login(identifier)` или `api.auth.register(identifier)` → уходит **POST** `/api/auth/login` или `/api/auth/register`. Этих маршрутов на бэкенде нет → **404**. Ошибка в коде игнорируется, выполняется переход на экран ввода кода.
2. **Экран подтверждения кода:** при нажатии «Подтвердить» бэкенд **не вызывается** — проверяется только локальный код (например, `0000` / `9999`), затем переход в приложение. Токен **нигде не запрашивается и не сохраняется**.
3. **Дальнейшие запросы:** `request.ts` подставляет в заголовки токен из `localStorage` по ключу `token`. Так как токен не был сохранён, запросы идут **без токена** → бэкенд возвращает **401** на `/api/auth/me`, ленту, профиль и т.д.

Итог: при выключенном моке вход «не завершается» — пользователь попадает в приложение без сессии, все запросы к API получают 401.

## 3. Необходимые изменения на фронте

Изменения в коде **ниже не внесены**; это спецификация для реализации.

### 3.1. Слой API (`src/integrations/`)

- **Типы (`api.types.ts`):**
  - Добавить тип ответа verify, например: `{ access_token: string; token_type: string; user: AppUser }`.
  - В интерфейс `AuthApi` добавить:
    - `sendCode(identifier: string): Promise<{ sent: boolean }>`;
    - `verify(identifier: string, code: string): Promise<{ access_token: string; user: AppUser }>`.
  - Методы `login` и `register` можно оставить для мок-режима или заменить первый шаг на вызов `sendCode` (см. ниже).

- **Реальный API (`api.ts`):**
  - Реализовать **sendCode**: **POST** `/auth/send-code` с телом `{ identifier }`. Путь при сборке уже с префиксом `/api` (в `request.ts`), итог: `/api/auth/send-code`.
  - Реализовать **verify**: **POST** `/auth/verify` с телом `{ identifier, code }`. Ответ — `access_token` и `user`. Сохранение токена в `localStorage` — в вызывающем коде (экран подтверждения кода) или внутри метода `verify` (записать в `localStorage` по ключу `token`).
  - Для шага «Получить код» использовать **sendCode** вместо `login`/`register` (см. п. 3.2 и 3.3).

- **Мок (`mockApi.ts`):**
  - Реализовать **sendCode**: возвращать `{ sent: true }` без вызова бэкенда.
  - Реализовать **verify**: возвращать объект вида `{ access_token: "...", user: ... }` (токен для мока может быть произвольным; при необходимости его тоже сохранять в `localStorage` для единообразия).

### 3.2. Экран входа (`src/pages/Login.tsx`)

- По кнопке «Получить код» вызывать **`api.auth.sendCode(value.trim())`** вместо `api.auth.login(value.trim())`.
- Обрабатывать ошибку ответа (например, показывать сообщение «Не удалось отправить код») вместо игнорирования в `catch`.
- После успешного ответа по-прежнему переходить на экран ввода кода с `navigate('/confirm-code', { state: { contact: value, mode: 'login' } })`.

### 3.3. Экран регистрации (`src/pages/Register.tsx`)

- По кнопке «Получить код» вызывать **`api.auth.sendCode(value.trim())`** вместо `api.auth.register(value.trim())`.
- Аналогично обрабатывать ошибки и при успехе переходить на экран кода с `state: { contact: value, mode: 'register' }`.

### 3.4. Экран подтверждения кода (`src/pages/ConfirmCode.tsx`)

- При нажатии «Подтвердить» вызывать **`api.auth.verify(contact, code)`**, где `contact` — из `location.state` (тот же identifier, что на шаге «Получить код»), `code` — введённый пользователем код.
- При успешном ответе:
  - Сохранить токен: **`localStorage.setItem('token', response.access_token)`** (ключ `token` уже используется в `request.ts` для заголовка `Authorization`).
  - Перейти в приложение: при `mode === 'register'` — на онбординг, иначе — на дерево (как сейчас).
- При ошибке (например, 400/401) показывать сообщение пользователю («Неверный код» или текст из ответа сервера), не переходить в приложение.

### 3.5. Обработка 401 при последующих запросах

- При получении 401 на любом запросе к API (истёк токен, невалидный токен) можно очищать `localStorage.token` и перенаправлять пользователя на экран входа, чтобы не оставлять его в «полузалогиненном» состоянии.

## 4. Краткое резюме

| Где | Что сделать |
|-----|-------------|
| Бэкенд | Менять не требуется: уже есть `send-code`, `verify`, `me` и JWT. |
| API-слой фронта | Добавить `sendCode` и `verify`, вызывать `/api/auth/send-code` и `/api/auth/verify`; после verify сохранять `access_token` в `localStorage` под ключом `token`. |
| Login / Register | Вызывать `sendCode` вместо `login`/`register` при «Получить код», обрабатывать ошибки. |
| ConfirmCode | При «Подтвердить» вызывать `verify`, сохранять токен и переходить в приложение; при ошибке показывать сообщение. |
| Общее | По желанию: при 401 на запросах — очистка токена и редирект на логин. |

После этих изменений авторизация при `VITE_USE_MOCK_API=false` будет проходить полностью через бэкенд: send-code → ввод кода → verify → сохранение токена → запросы с `Authorization: Bearer <token>`.
