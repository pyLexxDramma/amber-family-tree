MVP Angelo — схема “1 VM + managed DB + object storage”

Оценка под ожидания: 100 семей (~500 человек), медиахранилище ~3 TB.

Идея схемы: один сервер (VM) под фронтенд+backend, отдельно управляемая БД, медиа — в объектном хранилище.
Это самый простой и “неубиваемый” MVP: обновления одной VM, бэкапы БД и хранение медиа отдельно.
Компонент Что запускаем Рекомендуемый размер для MVP Комментарий
VM (Compute)
  • nginx (раздаёт фронтенд)
  • backend (API)
  • docker/compose (опционально)
4 vCPU / 8 GB RAM
SSD диск 80–120 GB
Диск нужен под ОС, логи, кэш, временные файлы, деплой.
Медиа на VM не храним — всё уходит в Object Storage.
Managed PostgreSQL
  • основные данные
  • публикации, связи, комментарии
  • ссылки на медиа (ключи/URL)
2 vCPU / 4–8 GB RAM
SSD 50–100 GB
Бэкапы включить, retention 7–14 дней.
Для 100 семей объём БД обычно небольшой, основное место занимает медиа.
При росте можно увеличить ресурсы без миграции приложения.
Object Storage
  • фото/видео/аудио
  • превью (thumbnail)
Квота/лимит: 4 TB (3 TB + запас)
Рекомендуемый запас: +25–35%.
Если план по продукту “100 GB на семью”, то формально 100×100 GB = 10 TB.
Но по вашим ожиданиям реальное потребление сейчас ~3 TB, поэтому ставим 4 TB и смотрим динамику.
CDN (опционально)
  • ускорение выдачи фото/видео
Можно не включать в MVP
Подключается позже, когда будет заметная нагрузка/география.
Мониторинг/логи
  • ошибки API (5xx)
  • latency, RPS
  • CPU/RAM/disk
Базовый мониторинг + алерты
Диск > 80%, 5xx, недоступность /api.
В MVP достаточно базовых графиков и уведомлений.
Ответы “по-простому” (для переписки):
  • Монолит: один backend-сервис + отдельные DB и S3-хранилище.
  • Для MVP: 1 VM (4×8) + Managed Postgres (2×4/8) + Object Storage (лимит 4 TB).
  • Почему так: проще деплой и поддержка, медиа не забивает диск сервера, бэкапы БД “из коробки”.
