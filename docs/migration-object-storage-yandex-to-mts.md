# Переезд объектного хранилища (Yandex Cloud → MTS Cloud и обратная совместимость)

Документ фиксирует **как** выполнять переезд медиа для Angelo, **риски**, **типичные трудности** и **оценку сроков**. Код приложения предполагается **без изменений**: достаточно S3-совместимого API (сейчас `boto3`, presigned `PUT`, переменные `S3_*` в backend).

---

## 1. Что именно переезжает

- **Файлы** в бакете (ключи вида `uploads/<user_id>/photo|video|audio|document/...`).
- **Конфигурация** на стороне приложения: `S3_ENDPOINT_URL`, `S3_PUBLIC_ENDPOINT_URL`, ключи, регион, имя бакета.
- **Инфраструктура** перед хранилищем: на Fornex часто Nginx проксирует `/angelo-media/` на MinIO; у облачного провайдера может быть **прямой HTTPS endpoint** без такого префикса — это влияет только на `S3_PUBLIC_ENDPOINT_URL` и на то, какие URL видит браузер при загрузке и отображении.

**База данных** хранит ключи или полные URL медиа; при смене только endpoint и бакета **миграция БД не обязательна**, если в БД лежат **относительные ключи**. Если когда-то сохраняли полные URL старого домена — потребуется **разовая правка** или редирект.

---

## 2. Рекомендуемый порядок работ (процесс)

### Этап A — Инвентаризация (1–3 рабочих дня)

- Зафиксировать: объём данных, число объектов, текущий бакет(ы), политики lifecycle (Standard / Cold / Ice), есть ли **публичный доступ** к объектам или только presigned.
- Проверить в проде: значения `S3_ENDPOINT_URL` / `S3_PUBLIC_ENDPOINT_URL`, как формируется ответ `POST /api/media/presign` (поля `upload_url`, `url`).
- Убедиться, что клиенты ходят на **HTTPS** для `PUT` (в коде фронта уже есть проверка на non-https upload URL в отдельных сценариях).

### Этап B — Пилот на MTS (3–10 рабочих дней)

- Создать **тестовый** бакет в MTS, выписать endpoint региона, ключи, регион для S3 API.
- Поднять **staging** или временный инстанс backend с env, указывающим на MTS; прогнить: регистрация, загрузка аватара, публикация с медиа, лента, профиль.
- Проверить **совместимость подписи** (SigV4): `PUT` из браузера по presigned URL, совпадение `Content-Type` с тем, что заложено в presign.

### Этап C — Копирование данных (зависит от объёма и канала)

- Инструменты: `rclone`, `aws s3 sync` с `--endpoint-url` для источника и приёмника, либо сервисные задачи копирования между бакетами.
- После копирования: сверка **количества объектов** и **суммарного размера**; выборочно — контрольные хеши/ETag.
- При **нулевом даунтайме**: стратегия *dual-write* (новые файлы в MTS) + фоновое догоняние старых ключей **или** заморозка записи на окно миграции — выбор зависит от допустимого простоя.

### Этап D — Переключение (cutover)

- Обновить **production** `.env` (или секреты оркестратора): endpoint, ключи, бакет, `S3_PUBLIC_ENDPOINT_URL` так, чтобы **presigned PUT** и **публичные URL** для чтения указывали на нужный хост (MTS или ваш домен/CDN поверх MTS).
- Перезапуск backend, смоук-тесты критических сценариев.
- Мониторинг ошибок 4xx/5xx на `/api/media/presign`, жалоб пользователей, метрики исходящего трафика и биллинга.

### Этап E — Стабилизация и отключение старого бакета

- Хранить read-fallback на старый бакет **только если** заранее заложили такую логику (в текущем коде отдельного fallback нет — все URL строятся из одного набора настроек).
- После периода наблюдения (например 1–2 недели) — удалить или архивировать старый бакет, отключить лишние платные опции на стороне Yandex.

## 3. Трудности и риски

| Риск | Суть | Как снижать |
|------|------|-------------|
| **Несовпадение URL** | Старые записи в БД с полным URL на домен Yandex/прокси; после смены endpoint ссылки ломаются. | Миграция полей в БД, редиректы, или хранить только ключи. |
| **Presigned PUT и CORS** | Браузер блокирует запрос или провайдер отклоняет подпись (заголовки, регион, clock skew). | Пилот из реального браузера; выровнять время на серверах; настроить CORS на бакете по доке MTS. |
| **Path-style vs virtual-hosted** | Разные провайдеры по-разному ожидают путь в URL. | Проверить фактические URL из `generate_presigned_url` и документацию MTS; при несовпадении — правка только конфигурации клиента boto3 (редко — код). |
| **`create_bucket` из приложения** | В managed-облаке создание бакета из API может быть запрещено или нежелательно. | Создавать бакет вручную в консоли; при ошибках head/create — не полагаться на автосоздание в проде. |
| **Стоимость операций и egress** | На Yandex после бесплатного лимита растёт счёт за исходящий интернет-трафик; на MTS нужно **подтвердить** условия по egress и операциям по актуальному договору/тарифу. | Считать TCO до переезда; заложить CDN при росте трафика. |
| **Долгая миграция 3–4 TB** | Ограничение скорости копирования, стоимость LIST/GET на стороне источника. | Копирование офф-пик, инкременты, контроль прогресса; по возможности перенос из того же дата-центра или через внутренние каналы провайдера (если доступно). |
| **Откат** | После переключения DNS/env откат требует снова указывать старые ключи и целостность данных в обоих бакетах. | Держать старый бакет read-only до финального решения; документировать точное состояние env на момент cutover. |

---

## 4. Оценка временных рамок (ориентиры)

Сроки сильно зависят от объёма данных, доступной полосы копирования и того, нужен ли **нулевой** даунтайм.

| Фаза | Ориентировочно |
|------|----------------|
| Инвентаризация и план | 1–3 дня |
| Пилот + исправление конфигов | 3–10 дней |
| Копирование **до ~1 TB** при стабильном канале | от нескольких часов до 2–5 дней (с учётом проверок) |
| Копирование **3–4 TB** | от нескольких дней до **2–4 недель** (параллельно с обычной работой команды) |
| Cutover и смоук-тесты | **2–8 часов** (окно обслуживания, если допустимо) |
| Наблюдение перед удалением старого бакета | **1–2 недели** |

Итого для проекта с **уже готовым** S3-клиентом и без миграции данных в БД: **от ~2 недель** в спокойном темпе до **1–2 месяцев**, если много данных и жёсткие требования к непрерывности.

---

## 5. Связь с текущим репозиторием Angelo

- Backend: переменные окружения `S3_ENDPOINT_URL`, `S3_PUBLIC_ENDPOINT_URL`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_REGION`, `S3_BUCKET`; эндпоинт `POST /api/media/presign` выдаёт presigned `PUT` и поле `url` для отображения.
- На Fornex типична схема: `S3_PUBLIC_ENDPOINT_URL=https://<домен>/angelo-media` и Nginx прокси на порт MinIO; у облачного провайдера публичный URL может быть **другим** — это настраивается **только env**, без обязательных правок логики, если API остаётся S3-совместимым.

---

## 6. Чеклист перед cutover

- [ ] Бакет в MTS создан, политика доступа и CORS проверены.
- [ ] Тестовый presigned `PUT` и `GET` с прод-подобного клиента успешны.
- [ ] Данные скопированы и сверены по объёму/выборке.
- [ ] Обновлены секреты и `S3_PUBLIC_ENDPOINT_URL`; сохранён бэкап старых значений env.
- [ ] План отката и ответственные на время переключения.
